Topic 01

Java Program Structure

The skeleton every Java file must follow. Get this wrong and nothing compiles.

📐
SYNTAX RECAP
Required OrderJava
// 1st — package (optional, but must be FIRST if used) package com.myapp; // 2nd — imports import java.util.Scanner; // 3rd — class (filename must match public class name) public class MyClass { // fields, methods, constructors // Entry point — exact signature required public static void main(String[] args) { System.out.println("Runs here"); } }
ElementRequired?Must Be
packageNoFirst statement if present
importNoAfter package, before class
Class nameYesMust match filename exactly (case-sensitive)
main signatureFor entry pointpublic static void main(String[] args)
🧠
COMMON BEGINNER MISTAKES
  • Naming file student.java but class is public class Student → compile error
  • Writing import before package → compile error
  • Writing void main() instead of public static void main(String[] args) → JVM won't find it
  • Thinking java.lang needs to be imported — it's auto-included

Java is fun Done
Step by step: print() writes without newline. So "Java", " is", " fun" all land on line 1. println(" fun") adds the newline. println() with no args prints just a blank line. Then "Done" on its own line.

❌ Compile Error: class Hello is public, should be declared in a file named Hello.java
Why: Java is case-sensitive. The file must be named Hello.java (capital H) to match public class Hello.
Will it compile?Q1.3

❌ Compile Error: package declaration must be first statement
Rule: Order is always: package → import → class. Never swap them.
Output?Q1.4
public class Nums { public static void main(String[] args) { System.out.println(1 + 2); System.out.println("1" + 2); System.out.println(1 + "2"); System.out.println("" + 1 + 2); } }
💭 Predict all 4 lines!
3 12 12 12
Line 1: int+int = 3. Line 2: String "1" + int 2 → String concatenation → "12". Line 3: int 1 + String "2" → "12". Line 4: "" + 1 → "1", then "1" + 2 → "12". The + operator becomes concatenation as soon as one side is a String.
Find the BugQ1.5
public class Runner { static void main(String[] args) { System.out.println("Running!"); } }
💭 Will the JVM run this? Why not?
Error: Main method not found in class Runner Please define the main method as: public static void main(String[] args)
Bug: Missing public keyword. The JVM looks for public static void main exactly. Removing public means the JVM cannot find the entry point. The class compiles fine, but running it fails.
⚠️ Exam Traps — Program Structure
  • A file can contain multiple classes but only ONE can be public, and it must match the filename
  • main(String args[]) and main(String[] args) are both valid — the bracket position doesn't matter
  • Forgetting static in main → compiles fine but throws error at runtime
  • java.lang.* is always imported — String, System, Math never need importing
Topic 02

Classes & Objects

Class = blueprint. Object = the real thing built from it. This is OOP's core.

📐
SYNTAX RECAP
Class AnatomyJava
public class Dog { // Blueprint // Fields (attributes) String name; int age; // Method (behavior) void bark() { System.out.println(name + " says: Woof!"); } } // Creating objects (instances) Dog d1 = new Dog(); // d1 is a reference on STACK pointing to heap Dog d2 = new Dog(); // separate object, own fields d1.name = "Rex"; // dot operator accesses fields d1.bark(); // dot operator calls methods

MEMORY VIEW
STACK (main method's local variables)
d1 →
0xA100
──────→
Dog{ name="Rex", age=0 }
d2 →
0xB200
──────→
Dog{ name=null, age=0 }
HEAP (objects live here)
KEY: d1 and d2 are REFERENCES (addresses), not the objects themselves. The actual data lives on the Heap.

🔍 Output Prediction

Output?Q2.1
class Car { String brand; int speed; } public class Main { public static void main(String[] args) { Car c = new Car(); System.out.println(c.brand); System.out.println(c.speed); } }
💭 Fields were never assigned. What prints?
null 0
Default values: Instance fields auto-initialize. String → null. int → 0. boolean → false. double → 0.0. This is ONLY for instance/static fields, NOT local variables!
Output?Q2.2 — Reference Sharing Trap
class Student { String name; } public class Main { public static void main(String[] args) { Student a = new Student(); a.name = "Abebe"; Student b = a; // THINK carefully! b.name = "Meron"; System.out.println(a.name); System.out.println(b.name); } }
💭 Does b = a copy the object or the reference?
Meron Meron
Memory: b = a copies the address (reference), NOT the object. Both a and b point to the SAME object on the heap. Changing b.name changes the shared object. So a.name is also "Meron".

Stack: a → 0xA100, b → 0xA100 (same!)
Heap: Student{name="Meron"} at 0xA100
Output?Q2.3
class Dog { String name = "Buddy"; void speak() { System.out.println(name + ": Woof"); } } public class Main { public static void main(String[] args) { Dog x = new Dog(); Dog y = new Dog(); y.name = "Max"; x.speak(); y.speak(); } }
💭 Two objects, separate fields — trace each one
Buddy: Woof Max: Woof
Unlike the previous question, x and y each got their own new Dog() call. Two separate objects on heap. Changing y.name only affects y's object.
Will it compile?Q2.4
public class Main { public static void main(String[] args) { Student s; System.out.println(s.name); } } class Student { String name; }
💭 Declared but never initialized with new. What happens?
❌ Compile Error: variable s might not have been initialized
Local variable s was declared but never assigned (new Student() was never called). Java forces you to initialize local variables before use. If s were an instance field, it would be null — but local variables have NO default.
Output?Q2.5 — Null Reference
class Phone { String model; } public class Main { public static void main(String[] args) { Phone p = null; System.out.println(p.model); } }
💭 p is null. What happens when you access its field?
❌ Runtime Error: NullPointerException (NullPointerException)
p is null — it doesn't point to any object. Trying to access p.model is like asking "what's in the room at address 0?" — there is no room. NullPointerException is thrown at RUNTIME (not compile time), because the compiler doesn't track null values.
⚠️ Exam Traps — Classes & Objects
  • b = a does NOT copy the object — it copies the reference address
  • Instance fields default to null/0/false. Local variables do NOT — they must be initialized
  • Accessing any member of a null reference → NullPointerException at runtime
  • Two different objects from the same class are completely independent
Topic 03

Object Instantiation

What exactly happens when you write new Dog()? Follow the memory step by step.

📐
SYNTAX + STEPS
Instantiation
Dog d = new Dog(); // Step 1: JVM allocates memory on the HEAP for a Dog object // Step 2: Fields initialized to defaults (null, 0, false) // Step 3: Constructor Dog() runs // Step 4: Address of heap object stored in d (on STACK)
  • 1
    Heap allocation: JVM finds free space on the heap and reserves it for the new object
  • 2
    Default init: all instance fields set to zero-equivalents (null, 0, 0.0, false)
  • 3
    Constructor runs: your initialization code executes, sets field values
  • 4
    Reference stored: the heap address is stored in the variable on the stack
Memory?Q3.1
Student s1 = new Student(); Student s2 = s1; Student s3 = new Student();
💭 How many Student objects exist on the heap?
2 objects on the heap
new was called TWICE. s1 and s2 both point to the first object. s3 points to the second. = s1 copies the reference, not a new object.
Output?Q3.2
class Employee { int id = 100; } public class Main { public static void main(String[] args) { Employee e1 = new Employee(); Employee e2 = new Employee(); e1.id = 200; System.out.println(e1.id + " " + e2.id); } }
💭 Each object initializes id=100. What after changing e1.id?
200 100
Two separate objects. e1.id = 200 only changes e1's own copy of id. e2.id stays 100 because they're independent objects.
Compile?Q3.3
public class Main { public static void main(String[] args) { int x; System.out.println(x); } }
💭 x is a local variable — does Java give it a default?
❌ Compile Error: variable x might not have been initialized
Local variables NEVER get default values. You must assign them before use. Only instance and static fields get defaults automatically.

Output?Q4.1
class BankAccount { double balance; void deposit(double amt) { balance += amt; } } public class Main { public static void main(String[] args) { BankAccount acc1 = new BankAccount(); BankAccount acc2 = new BankAccount(); acc1.deposit(500); acc2.deposit(200); acc1.deposit(300); System.out.println(acc1.balance); System.out.println(acc2.balance); } }
💭 Trace each account's balance separately
800.0 200.0
acc1 gets 500+300=800. acc2 gets 200. Instance variables are fully independent per object. Calling deposit on acc1 never affects acc2.
Find the BugQ4.2
class Employee { String name; double salary; void raiseSalary(double pct) { double salary = salary * (1 + pct / 100); } }
💭 Will this update the employee's salary?
❌ Bug: local variable shadows instance variable + compile error
double salary = salary * ... declares a LOCAL variable named salary which shadows the instance field. This local variable is used before initialization → compile error. Fix: remove double keyword: salary = salary * (1 + pct/100); — or use this.salary for clarity.
Output?Q4.3
class Counter { int count; void increment() { int count = this.count + 1; // local variable! } } public class Main { public static void main(String[] args) { Counter c = new Counter(); c.increment(); c.increment(); System.out.println(c.count); } }
💭 Notice: increment creates a local int count, not updating the field
0
Inside increment(), int count = this.count + 1 creates a LOCAL variable and discards it. The instance field this.count is never modified. Fix: this.count = this.count + 1; or simply count++;

Rex: 4

Topic 06

Static Variables

One copy shared by ALL objects of the class. Lives in Method Area, not heap.

Instance Variable

  • Each object has its OWN copy
  • Lives on the HEAP inside each object
  • Accessed via object.field
  • Created when object is created
  • Destroyed when object is GC'd

Static Variable

  • ONE copy shared by ALL objects
  • Lives in METHOD AREA
  • Accessed via ClassName.field
  • Created when class is loaded
  • Destroyed when program ends
Pattern: Object Counter
class Student { String name; // instance — per object static int count = 0; // static — shared by all Student(String name) { this.name = name; count++; // every new student increments the shared counter } }
Output?Q6.1 — The Classic
class Student { static int count = 0; Student() { count++; } } public class Main { public static void main(String[] args) { Student s1 = new Student(); Student s2 = new Student(); Student s3 = new Student(); System.out.println(Student.count); System.out.println(s1.count); // valid but bad style } }
💭 count is static. How many times did the constructor run?
3 3
Static count is shared. Each new Student() calls the constructor, incrementing the SAME count. After 3 objects: count=3. s1.count and Student.count both access the same memory location.
Output?Q6.2 — Mutation Trap
class Car { static int maxSpeed = 120; int currentSpeed = 0; } public class Main { public static void main(String[] args) { Car c1 = new Car(); Car c2 = new Car(); c1.maxSpeed = 200; // bad style, but legal c1.currentSpeed = 50; System.out.println(c2.maxSpeed); System.out.println(c2.currentSpeed); } }
💭 Changing c1.maxSpeed affects c2 — why?
200 0
maxSpeed is static — changing it via c1 changes THE ONLY COPY, which c2 also sees. currentSpeed is instance — c1's change only affects c1's object. c2.currentSpeed stays 0.
Output?Q6.3
class Phone { static String brand = "Samsung"; String model; Phone(String model) { this.model = model; } } public class Main { public static void main(String[] args) { Phone p1 = new Phone("A52"); Phone p2 = new Phone("S21"); Phone.brand = "Apple"; System.out.println(p1.brand + " " + p1.model); System.out.println(p2.brand + " " + p2.model); } }
💭 brand is changed after creating both phones
Apple A52 Apple S21
brand is static — changing it changes it for EVERYONE. Both p1 and p2 now see "Apple". model is instance, so p1 keeps "A52" and p2 keeps "S21".
Output?Q6.4
class Test { static int x; } public class Main { public static void main(String[] args) { System.out.println(Test.x); } }
💭 Static field, never assigned. What's its default?
0
Static fields, like instance fields, are auto-initialized. int static → 0. No NullPointerException, no error. This is safe to read.
Find the BugQ6.5
class Employee { static int headcount = 0; String name; Employee(String n) { name=n; headcount++; } void fire() { headcount--; } // does this work? static void getHeadcount() { System.out.println(name); } // ?? }
💭 Which line has a bug and why?
❌ Compile Error on getHeadcount(): non-static variable name cannot be referenced from a static context
getHeadcount() is a static method. Static methods don't have this — they don't know which object's name to use. Fix: print headcount instead of name: System.out.println(headcount);. The fire() method is fine — instance methods CAN access static fields.
⚠️ Exam Traps — Static Variables
  • Accessing static field via object ref (obj.staticField) is LEGAL but bad style — use ClassName.field
  • Static fields are initialized ONCE when the class loads, before any object is created
  • Static method CANNOT access instance variables (no this exists in static context)
  • Instance method CAN access static variables (it has access to everything)
Topic 07

Static Methods

Methods that belong to the class, not any object. No this. No instance access.

Instance Method

  • Needs an object to call
  • Has implicit this
  • Can access instance fields
  • Can access static fields
  • Called: obj.method()

Static Method

  • No object needed
  • No this — doesn't know which object
  • CANNOT access instance fields directly
  • Can access static fields
  • Called: ClassName.method()
Compile?Q7.1
class MathHelper { int value = 10; static int doubleIt(int n) { System.out.println(value); // accessing instance field in static! return n * 2; } }
💭 Can a static method use an instance field?
❌ Compile Error: non-static variable value cannot be referenced from a static context
Static methods are class-level. They execute without any object. If you try to access value, Java would ask "which object's value?" — but there's no object. Solution: either make value static, or pass it as a parameter, or create an object inside the method.
Output?Q7.2 — null reference static call
class Dog { static void bark() { System.out.println("Woof!"); } } public class Main { public static void main(String[] args) { Dog d = null; d.bark(); // d is null! } }
💭 d is null. Will calling a static method via null crash?
Woof!
Surprise! This works. When Java sees d.bark(), it resolves bark at compile time to Dog.bark() because it's static. The value of d at runtime is irrelevant — it's never actually dereferenced. No NullPointerException! This is a famous exam trap.
Output?Q7.3
class Calculator { static int ops = 0; static int add(int a, int b) { ops++; return a+b; } static int mul(int a, int b) { ops++; return a*b; } } public class Main { public static void main(String[] args) { System.out.println(Calculator.add(3,4)); System.out.println(Calculator.mul(2,5)); System.out.println(Calculator.ops); } }
💭 Track ops counter and return values
7 10 2
add(3,4)=7, ops becomes 1. mul(2,5)=10, ops becomes 2. Final ops=2.

Output?Q8.1
class Car { String brand; Car(String brand) { this.brand = brand; System.out.println("Car created: " + brand); } } public class Main { public static void main(String[] args) { Car c1 = new Car("Toyota"); Car c2 = new Car("BMW"); } }
💭 Constructor runs once per new. How many times?
Car created: Toyota Car created: BMW
Constructor runs once for each new. Two new calls = constructor runs twice.
Compile?Q8.2 — The Void Trap
class Student { String name; void Student(String name) { // Is this a constructor? this.name = name; } } public class Main { public static void main(String[] args) { Student s = new Student("Ali"); System.out.println(s.name); } }
💭 void Student() — is this a constructor?
❌ Compile Error: constructor Student in class Student cannot be applied to given arguments
void Student(String name) is NOT a constructor — it's a regular method. A constructor has NO return type whatsoever. Since no real constructor exists, Java provides a no-arg default. But new Student("Ali") looks for a constructor taking a String — which doesn't exist. Result: compile error.
Compile?Q8.3 — No Default Constructor
class Dog { String name; Dog(String name) { this.name = name; } } public class Main { public static void main(String[] args) { Dog d = new Dog(); // No-arg constructor } }
💭 You wrote a parameterized constructor. Does Java still give you a default?
❌ Compile Error: constructor Dog in class Dog cannot be applied to given arguments
Java ONLY provides a default no-arg constructor if you write ZERO constructors. The moment you write ANY constructor, the free default is gone. new Dog() fails because there's no no-arg constructor. Fix: add Dog() {} or use new Dog("Rex").
Topic 09

Constructor Types

Default, Parameterized, Copy, and Chained (this()). Know all four deeply.

All 4 TypesJava
class Student { String name; int age; double gpa; // TYPE 1: Default (no args) Student() { this("Unknown", 18, 0.0); } // chains to full! // TYPE 2: Parameterized Student(String name, int age, double gpa) { this.name=name; this.age=age; this.gpa=gpa; } // TYPE 3: Copy constructor Student(Student other) { this(other.name, other.age, other.gpa); } }
Output?Q9.1 — Chaining Trace
class BankAccount { double balance; String owner; BankAccount() { this("Anonymous"); System.out.println("A"); } BankAccount(String owner) { this(owner, 0.0); System.out.println("B"); } BankAccount(String owner, double balance) { this.owner=owner; this.balance=balance; System.out.println("C"); } } public class Main { public static void main(String[] args) { new BankAccount(); } }
💭 Trace the chain: no-arg → String → String+double. What order do prints happen?
C B A
this() jumps to another constructor. Execution goes DEEP first, then unwinds. No-arg calls this("Anonymous") → execution pauses at A. String constructor calls this(owner, 0.0) → pauses at B. Full constructor runs → prints C. Returns to String constructor → prints B. Returns to no-arg → prints A. Like function call stack unwinding!
Output?Q9.2 — Copy vs Reference
class Dog { String name; Dog(String n) { name=n; } Dog(Dog other) { name=other.name; } // copy constructor } public class Main { public static void main(String[] args) { Dog d1 = new Dog("Rex"); Dog d2 = new Dog(d1); // copy constructor Dog d3 = d1; // reference copy d1.name = "Max"; System.out.println(d1.name); System.out.println(d2.name); System.out.println(d3.name); } }
💭 d2 uses copy constructor, d3 is reference. What changes when d1.name changes?
Max Rex Max
d2 was created with the copy constructor — it's a SEPARATE object with its own name field. d1.name changing doesn't affect d2.name. But d3 = d1 is just a reference copy — d3 points to THE SAME object as d1. So d3.name reflects the change to "Max".
Compile?Q9.3 — this() position
class Car { String brand; Car() { System.out.println("Creating"); // line 1 this("Default"); // line 2 } Car(String brand) { this.brand=brand; } }
💭 this() is on line 2, not line 1. Is that okay?
❌ Compile Error: call to this must be first statement in constructor
this() MUST be the absolute first statement in a constructor. You cannot put any code before it. Move this("Default") to line 1 and the println after it.
Topic 10

The this Keyword

The most misunderstood keyword. Three uses. Learn them all.

⚡
THREE USES OF this
UseSyntaxPurpose
1. Field disambiguationthis.name = nameDistinguish instance field from local param
2. Constructor chainingthis(args)Call another constructor from same class
3. Pass current objectsomeMethod(this)Send the current object as an argument
Output?Q10.1 — The Shadowing Bug
class Student { String name; int age; Student(String name, int age) { name = name; // BUG: assigns param to itself age = age; // BUG: same problem } } public class Main { public static void main(String[] args) { Student s = new Student("Ali", 20); System.out.println(s.name); System.out.println(s.age); } }
💭 Without this.name = name, what actually happens?
null 0
name = name assigns the PARAMETER to itself — it's a no-op. The INSTANCE FIELD this.name is never touched, so it stays null. Same for age. The fix is this.name = name; this.age = age;
Output?Q10.2
class Car { String brand; int year; Car(String brand) { this(brand, 2020); // chain to 2-arg constructor System.out.println("1-arg done"); } Car(String brand, int year) { this.brand=brand; this.year=year; System.out.println("2-arg done"); } } public class Main { public static void main(String[] args) { new Car("Toyota"); } }
💭 Which constructor prints first?
2-arg done 1-arg done
this(brand, 2020) jumps to the 2-arg constructor. That runs completely → "2-arg done". Then returns to 1-arg constructor which continues → "1-arg done". The code AFTER this() runs after the chained constructor finishes.
Output?Q10.3 — this in static context
class Dog { String name = "Rex"; static void show() { System.out.println(this.name); } }
💭 Can this be used in a static method?
❌ Compile Error: non-static variable this cannot be referenced from a static context
this refers to the current object. In a static method, no object exists — the method belongs to the class, not any object. So this has no meaning there. Java refuses to compile it.
Output?Q10.4 — Method returns this
class Builder { String result = ""; Builder add(String s) { result += s; return this; // return the same object } } public class Main { public static void main(String[] args) { Builder b = new Builder(); b.add("Hello").add(" World").add("!"); System.out.println(b.result); } }
💭 Returning this enables method chaining. What builds up?
Hello World!
Each add() appends to result and returns this (the same Builder object). So .add("Hello").add(" World").add("!") chains on the SAME object. Final result = "Hello World!". This pattern is called Method Chaining / Fluent API.
Topic 11

Static Block vs Instance Block

The most common execution-order exam trap in Java.

⚡
EXECUTION ORDER RULE
Golden Rule

Static block → runs ONCE when class is first loaded by JVM (before main!).
Instance block → runs before constructor, every time an object is created.
Constructor → runs after instance block, every time an object is created.

Static BlockInstance BlockConstructor
Runs whenClass loadsPer objectPer object
How many timesOnce onlyEvery newEvery new
Order1st2nd3rd
Can access instance members?NoYesYes
Output? — The ClassicQ11.1
class Dog { static { System.out.println("Static"); } { System.out.println("Instance"); } Dog() { System.out.println("Constructor"); } } public class Main { public static void main(String[] args) { System.out.println("main starts"); new Dog(); new Dog(); } }
💭 Note: class Dog is loaded BEFORE main starts. What order?
Static main starts Instance Constructor Instance Constructor
Dog class loads when JVM sees it referenced → Static block fires BEFORE main() even prints. Then main prints. Each new Dog() triggers: Instance block first, then Constructor. Static block never repeats.
Output?Q11.2 — Static field init order
class Config { static int x = compute(); static { System.out.println("Block x="+x); } static int compute() { System.out.println("compute()"); return 42; } } public class Main { public static void main(String[] args) { System.out.println("main"); new Config(); } }
💭 Static field initializer runs before static block. Track order carefully.
compute() Block x=42 main
When Config class loads: static field x = compute() runs first → "compute()". Then static block → "Block x=42". Then main starts → "main". new Config() creates an object but there are no instance blocks or custom constructor, so nothing more prints.

Method Overloading

Output?Q16.1
class Printer { void print(int x) { System.out.println("int: "+x); } void print(double x) { System.out.println("double: "+x); } void print(String x) { System.out.println("String: "+x); } void print(int x, int y) { System.out.println("2ints: "+x+","+y); } } public class Main { public static void main(String[] args) { Printer p = new Printer(); p.print(5); p.print(5.0); p.print("hi"); p.print(1,2); } }
💭 Match each call to the best signature
int: 5 double: 5.0 String: hi 2ints: 1,2
Java picks the most specific matching method at compile time. 5 is int literal → print(int). 5.0 is double literal → print(double). "hi" is String → print(String). Two ints → print(int,int).
Output?Q16.2 — Type Promotion
class Calc { void show(int x) { System.out.println("int"); } void show(long x) { System.out.println("long"); } } public class Main { public static void main(String[] args) { Calc c = new Calc(); byte b = 5; c.show(b); } }
💭 b is byte. No show(byte). Which method gets called?
int
Widening promotion: byte → short → int → long → float → double. Java promotes to the NEAREST wider type first. byte promotes to int before long. So show(int) is chosen.
Compile?Q16.3 — Return type only
class Test { int getValue() { return 1; } double getValue() { return 1.0; } }
💭 Only return type differs. Is this valid overloading?
❌ Compile Error: method getValue() is already defined in class Test
Return type is NOT part of the method signature for overloading purposes. Java can't tell which to call if you write getValue() — are you expecting int or double? It's ambiguous. Not allowed.
Output?Q16.4 — Ambiguous call
class Math { void compute(int a, double b) { System.out.println("A"); } void compute(double a, int b) { System.out.println("B"); } } public class Main { public static void main(String[] args) { new Math().compute(5, 5); // two ints } }
💭 Two int args. Can Java promote either one to double to match?
❌ Compile Error: reference to compute is ambiguous
Java could promote the first int to double (→ calls compute(double,int)) OR the second int to double (→ calls compute(int,double)). Both are equally valid. Java gives up → ambiguous call error.

Topic 18

Argument Passing

Java is ALWAYS pass-by-value. For primitives: value copied. For objects: reference copied.

⚡
THE GOLDEN RULE

Primitive argument

  • A COPY of the value is passed
  • Original variable is NEVER changed
  • Method works on its own copy

Object argument

  • A COPY of the reference is passed
  • Method can modify the object's fields
  • But can't make caller's variable point elsewhere
Output?Q18.1 — Primitive
public class Main { static void triple(int x) { x = x * 3; System.out.println("Inside: " + x); } public static void main(String[] args) { int n = 10; triple(n); System.out.println("Outside: " + n); } }
💭 Does triple() modify the original n?
Inside: 30 Outside: 10
n=10 is passed by value. triple() gets a COPY named x. x=30 inside the method. But n in main never changes. Primitives = always safe from modification.
Output?Q18.2 — Object argument
class Dog { String name; } public class Main { static void rename(Dog d) { d.name = "Max"; // modifies the object } public static void main(String[] args) { Dog myDog = new Dog(); myDog.name = "Rex"; rename(myDog); System.out.println(myDog.name); } }
💭 rename() modifies d.name. Does it affect myDog?
Max
myDog's reference (address 0xA100) is copied into d. d still points to the SAME Dog object. Modifying d.name modifies the actual object, which myDog also sees. The object was changed — not the reference variable.
Output?Q18.3 — Reassigning the reference
class Dog { String name; } public class Main { static void replaceWith(Dog d) { d = new Dog(); // d now points to NEW object d.name = "Max"; } public static void main(String[] args) { Dog myDog = new Dog(); myDog.name = "Rex"; replaceWith(myDog); System.out.println(myDog.name); } }
💭 replaceWith makes d point to a NEW object. Does myDog change?
Rex
myDog's reference was COPIED into d. d = new Dog() makes d point to a new object — but this only changes the LOCAL copy d. myDog in main still points to the original Dog with name "Rex". You cannot make a caller's variable point to a different object through a method parameter.

GC?Q19.2
public class Main { static void createDog() { Dog temp = new Dog(); temp.name = "Fluffy"; // method ends here } public static void main(String[] args) { createDog(); // Is the Dog still alive? } }
💭 temp is a local variable. What happens when the method returns?
The Dog object becomes GC-eligible when createDog() returns
When createDog() exits, its stack frame is destroyed. temp (the only reference to the Dog) disappears. The Dog on the heap has no more references → GC-eligible. The object is NOT immediately destroyed — GC runs at its own schedule.

FALSE — System.gc() is just a HINT. The JVM may or may not honor it.
System.gc() suggests that garbage collection should run, but the JVM is free to ignore this hint. There is NO way to force GC to run at a specific time in Java. This is by design — GC is fully managed by the JVM.

⚠️ Exam Traps — Garbage Collection
  • System.gc() is a HINT, not a command. GC timing is non-deterministic.
  • An object becomes eligible for GC when NO MORE references point to it — not when you set one variable to null
  • Circular references (A points to B, B points to A) with no external reference → both are GC-eligible. Java's GC handles cycles.
  • You cannot resurrect an object once it has been GC'd
  • finalize() is deprecated — never rely on it for cleanup
